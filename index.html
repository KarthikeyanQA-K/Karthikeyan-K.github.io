public class LoginPage {
    private final WebDriver driver;
    private final By email = By.id("email");
    private final By password = By.id("password");
    private final By submit = By.cssSelector("[data-test=login]");

    public LoginPage(WebDriver driver) { this.driver = driver; }

    public DashboardPage loginAs(User user) {
        driver.findElement(email).sendKeys(user.email());
        driver.findElement(password).sendKeys(user.password());
        driver.findElement(submit).click();
        return new DashboardPage(driver);
    }
}

Approach

How I approach a build

The same four steps whether I'm joining a mature suite or writing the first test on a new product.

1

Find the risk first

Map the flows where a bug costs money, data or trust. Those get depth; the rest get a smoke check.

2

Test manually, then automate

A script can only assert what someone already understood. I explore the flow first, then encode it.

3

Automate what's stable

Page objects, data-driven inputs, no hard waits. Flaky tests get fixed or removed — never ignored.

4

Put it in the pipeline

A suite nobody runs is documentation. Jenkins runs it on every build and reports where it broke.
